# Production Pipeline Audit

## Executive Summary

**The repository uses a fully manual, human-operated prompt workflow to produce all published pages. There is no automated pipeline.**

The documented "Agent Pipeline" (Opportunity Discovery → ORA → Research Compiler → Editorial Builder → Editorial QA → Publisher) does NOT produce any published page. It is a documented future design. Every one of the 30 published pages was generated by copying a prompt (`docs/PRODUCTION-MASTER-PROMPT.md`, `docs/BLOG-MASTER-PROMPT.md`, or `docs/ROUNDUP-MASTER-PROMPT.md`) into a Claude conversation, feeding in a research package, and manually placing the generated `.astro` file into the repository.

**Confidence: High.** The evidence is conclusive across four independent dimensions:
1. Every published page is a self-contained `.astro` file with zero imports, contradicting the Agent Editorial Builder's component-based architecture
2. The shared components the Agent pipeline requires (`OlspLayout`, `QuoteBanner`, `StandardCta`, `AuthorBox`, etc.) do not exist in the codebase
3. The three `docs/*-MASTER-PROMPT.md` files are explicit "copy this into Claude" instructions — they are not agent prompts
4. Git history shows the agent pipeline was documented *after* all published pages were already produced

## Evidence

### Evidence 1 — Zero imports across all 30 article pages

Every published page (14 reviews, 22 blog articles, 1 roundup) is a fully self-contained `.astro` file. None contain any `import` statement:

```
src/pages/reviews/ — 14 files, 0 imports
src/pages/blog/ — 22 files, 0 imports
src/pages/roundups/ — 1 file, 0 imports
```

Only the infrastructure pages (`index.astro`, `authors/jarmo-halonen.astro`) use imports — they are not article pages.

**Why this matters:** The Agent Editorial Builder prompt (`agents/editorial-builder/PROMPT.md` v2.1) describes an architecture where the Builder outputs an Astro content block wrapped in `<OlspLayout>` with shared components (`QuoteBanner`, `StandardCta`, `AuthorBox`, `ScoreBars`, `Quiz`, `ComparisonTable`, `ProsCons`, `SvgDiagram`, `VideoEmbed`). If any published page had been produced by this agent, it would contain `import OlspLayout from...` or component imports. None do.

## Current Production Flow

The actual flow used today is:

```
Research (manually compiled: Opportunity Brief or Knowledge Asset or ad-hoc notes)
    ↓
Human selects production prompt (docs/PRODUCTION-MASTER-PROMPT.md or BLOG-MASTER-PROMPT.md or ROUNDUP-MASTER-PROMPT.md)
    ↓
Human copy-pastes prompt + research + inputs into Claude (or similar LLM)
    ↓
LLM generates complete self-contained .astro file
    ↓
Human reviews, places file in correct src/pages/ directory
    ↓
Human runs astro build to verify
    ↓
Human commits and pushes
```

**Execution changes from implemented production code to placeholder at every step.** There is no automated stage. Every stage depends on human manual work.

## Recommended Next Task

**Audit and reconcile the Agent Editorial Builder PROMPT.md's architecture assumptions against the actual production codebase.**

The single highest-uncertainty item is whether the Agent Editorial Builder prompt describes a system that can actually be built. The prompt references components (`OlspLayout`, `QuoteBanner`, `StandardCta`, `AuthorBox`, etc.) that don't exist, and an architecture (shared layout + imported components) that is the opposite of the self-contained production pages.

The smallest task that reduces architectural uncertainty:

**1. Identify every missing component and architecture element referenced in `agents/editorial-builder/PROMPT.md` that doesn't exist in the codebase.**
**2. List them — and nothing else.**

This is pure inspection: read the PROMPT.md, compare every component name, import path, and architectural assumption against `src/layouts/`, `src/components/`, and every published `.astro` page. Return a gap list. No code is written, no decision is made about which direction to go (migrate production to components vs rewrite the prompt to match production).

This task takes ~15 minutes, produces a concrete artifact, and directly enables the next architectural decision: whether to build the component system the prompt assumes, or rewrite the prompt to match the actual self-contained production architecture.

**Why not "build the components" or "rewrite the prompt" as the first step?** Because we don't yet know which direction is correct. Building the components would create a system that matches the prompt but diverges from production reality. Rewriting the prompt to match self-contained production would abandon the v2.1 architecture freeze. The gap list enables an informed decision rather than a premature one.
